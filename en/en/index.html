    Product Introduction¶

    gpuctl is an AI compute scheduling platform for ML engineers, designed to significantly lower the barrier to GPU computing.

    Through declarative YAML configuration and simple CLI commands, ML engineers can efficiently submit and manage AI training and inference tasks without mastering complex Kubernetes infrastructure.

    Core Pain Points Solved¶

    Pain Point Impact gpuctl Solution
    😰 High K8s Learning Curve
    Pod, Deployment, Service concepts    		 ML engineers need weeks to learn Kubernetes concepts like PodSpec, ResourceRequirements, VolumeMounts. Submitting tasks requires 100+ lines of YAML across multiple resources (Secret, ConfigMap, Job). Declarative YAML with familiar fields
    Use kind, job.name, resources.gpu fields known to ML engineers. Submit training tasks with just 15-20 lines of config. No need to understand underlying K8s resources.
    😤 Complex GPU Environment Setup
    Drivers, CUDA, NCCL dependencies    		 Each new environment requires manual installation of GPU drivers, CUDA Toolkit, cuDNN, NCCL configuration, plus DeepSpeed, VLLM frameworks. Version conflicts waste hours or days. Pre-installed images with auto-injected env vars
    Official images with DeepSpeed, VLLM, LlamaFactory pre-installed. Platform auto-injects NCCL_SOCKET_IFNAME, MASTER_ADDR, WORLD_SIZE. No manual distributed training configuration needed.
    😫 Multi-team GPU Contention
    No resource isolation    		 Training, inference, and experimentation tasks run mixed in the same cluster. No isolation means critical tasks get preempted by low-priority jobs. One team's large model training can occupy all GPUs. Resource pool isolation + quota management
    Divide cluster into training pool, inference pool, development pool for physical isolation. Set CPU/Memory/GPU quotas per namespace. Auto-reject when exceeded to ensure fair resource usage.
    😵 Complex Multi-GPU Training
    NCCL, DeepSpeed parameters    		 Multi-GPU training requires manual NCCL environment setup, DeepSpeed hostfile, PyTorch launch parameters. Need to understand process groups and communication backends. Configuration errors cause training hangs or inefficiency. Just declare GPU count
    Declare `resources.gpu: 4` in YAML. Platform auto-generates DeepSpeed config, injects NCCL environment variables, and initializes process groups. ML engineers don't need to understand underlying distributed details.
    😵‍💫 Tedious Task Status Checking
    Random pod names hard to remember    		 Using kubectl requires remembering randomly generated pod names like `training-job-7d9f4b8c5-x2mnp`. First get jobs, then get pods, then describe pod. Pod restarts change names, requiring re-finding. Tedious and time-consuming. Use task name directly, auto-track pod changes
    `gpuctl get jobs` shows all task statuses. `gpuctl logs job-name` views logs directly with multi-replica aggregation. Auto-tracks pod rebuilds and status changes. Just remember the task name you defined.

    System Architecture¶

    Supported Task Types¶

    For LLM fine-tuning (LlamaFactory + DeepSpeed), single-node multi-GPU training. Converts to K8s Job.

    Key Benefits: Auto NCCL configuration, checkpoint save/restore, automatic resource release after training completes.

    kind: training
version: v0.1
job:
  name: qwen2-7b-sft
  priority: high
environment:
  image: registry.example.com/llama-factory-deepspeed:v0.8.0
  command: ["llama-factory-cli", "train", "--stage", "sft", "--model_name_or_path", "/models/qwen2-7b"]
resources:
  pool: training-pool
  gpu: 4
  gpu-type: A100-100G
  cpu: 32
  memory: 128Gi

    For VLLM inference service deployment with auto-scaling. Converts to K8s Deployment + Service + HPA.

    Key Benefits: Auto NodePort generation, built-in health checks, multi-replica load balancing.

    kind: inference
version: v0.1
job:
  name: llama3-8b-inference
environment:
  image: vllm/vllm-serving:v0.5.0
  command: ["python", "-m", "vllm.entrypoints.openai.api_server"]
service:
  replicas: 2
  port: 8000
resources:
  pool: inference-pool
  gpu: 1
  cpu: 8
  memory: 32Gi

    For interactive debugging with JupyterLab environment. Converts to K8s StatefulSet + Service.

    Key Benefits: One-click launch, auto-generated access links, support for custom images and passwords.

    kind: notebook
version: v0.1
job:
  name: dev-notebook
environment:
  image: registry.example.com/jupyter-ai:v1.0
service:
  port: 8888
resources:
  pool: dev-pool
  gpu: 1
  cpu: 8
  memory: 32Gi

    For general CPU services (nginx, redis, etc.). Converts to K8s Deployment + Service.

    Key Benefits: Multi-replica deployment, automatic load balancing, suitable for general compute and batch processing.

    kind: compute
version: v0.1
job:
  name: test-nginx
environment:
  image: nginx:latest
service:
  replicas: 2
  port: 80
resources:
  pool: test-pool
  gpu: 0
  cpu: 2
  memory: 4Gi

    Quick Start¶

    # 1. Install CLI
pip install gpuctl

# 2. Submit LLM fine-tuning task (4x A100)
cat > training.yaml << 'EOF'
kind: training
version: v0.1
job:
  name: qwen2-7b-sft
environment:
  image: llama-factory:latest
  command: ["llamafactory-cli", "train", "--stage", "sft"]
resources:
  pool: training-pool
  gpu: 4
  cpu: 32
  memory: 128Gi
EOF

gpuctl create -f training.yaml

# 3. Check task status
gpuctl get jobs

# 4. View logs in real-time
gpuctl logs qwen2-7b-sft -f

# 5. Delete task
gpuctl delete job qwen2-7b-sft
